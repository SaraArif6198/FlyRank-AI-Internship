1  The Real Task — From My FL-01 Audit

FL-01 Target Task 1 · Automating SQL migration scripts for the DG Cement Work Tracker

Classified "Collaborate with AI" in my Week 1 audit: I define the target schema requirements, AI suggests correct syntax and constraints for MySQL. The concrete job here: add audit-log capability to the live database that pays 50+ workers — where an untraceable change to attendance or payroll can become a labour dispute. FL-01 already defined what success means:

(a) runs without syntax errors (b) all existing production rows preserved (c) < 500 ms on a 10,000-row database

2  The Iteration Log — Six Versions, One Technique Each

V0 · The Naive Prompt

what I'd have typed a month ago
Prompt
Write a SQL migration to add an audit log table to my database.
Output — representative excerpt
CREATE TABLE audit_log ( id SERIAL PRIMARY KEY, -- PostgreSQL, not MySQL table_name VARCHAR(255), action VARCHAR(50), user_id INTEGER, timestamp TIMESTAMP DEFAULT NOW() ); "…you may want to adjust the columns based on your needs…"
Note — what the output didWrong database. SERIAL and NOW() are PostgreSQL — my system is MySQL. Generic columns tied to nothing in my schema, no rollback, and "adjust based on your needs" hands the work back to me. Scores 0 of 3 against my FL-01 criteria — it wouldn't even parse. Next: it doesn't know what kind of engineer should be answering → role assignment.

V1 · Technique: Role Assignment

+ senior MySQL 8 DBA persona
Prompt — change highlighted
You are a senior MySQL 8 DBA who writes migrations for live production systems and treats data loss as a career-ending event. Write a SQL migration to add an audit log table to my database.
Output — representative excerpt
-- Run inside a transaction where possible; take a backup first. CREATE TABLE audit_log ( id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT PRIMARY KEY, table_name VARCHAR(64) NOT NULL, action ENUM('INSERT','UPDATE','DELETE') NOT NULL, changed_by VARCHAR(64) NOT NULL, changed_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
Note — what improved in the outputThe dialect fixed itself without me listing a single syntax rule — AUTO_INCREMENT, InnoDB, utf8mb4, an ENUM, plus an unprompted "take a backup first." The role carried a whole set of conventions I never spelled out. Still failed: it's auditing a database it has never seen — no link to my tables, no old/new-value capture, which is the whole point for dispute resolution. Next: context & motivation.

V2 · Technique: Context & Motivation

+ live schema, the why, hard constraints
Prompt — change highlighted
[V1 prompt, plus:] Context — the live schema (excerpts): workers(id PK, name, cnic, site_id, daily_rate, created_at) attendance(id PK, worker_id FK→workers.id, work_date, status, hours, marked_by) payroll_periods(id PK, period_start, period_end, locked) Production database paying 50+ construction workers; ~10,000 rows in attendance. Why this matters: wages are calculated from attendance on a 26-day cycle — if a record changes and nobody can prove who changed it and what it was before, that's a labour dispute. Every change to attendance and payroll_periods must be traceable: who, when, old value, new value. Constraints: all existing rows preserved; runs in under 500 ms on 10k rows.
Output — representative excerpt
CREATE TABLE audit_log ( id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT PRIMARY KEY, table_name ENUM('attendance','payroll_periods') NOT NULL, record_id BIGINT UNSIGNED NOT NULL, action ENUM('INSERT','UPDATE','DELETE') NOT NULL, old_values JSON NULL, new_values JSON NULL, changed_by VARCHAR(64) NOT NULL, changed_at TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3), INDEX idx_audit_record (table_name, record_id) ) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
Note — what improved in the outputIt designed for my problem: the ENUM locks auditing to exactly the two dispute-sensitive tables, and old_values/new_values JSON appeared because I explained the labour-dispute motivation — I never asked for those columns; the why produced them. The index anticipates "show me this record's history." Still failed: a SQL blob in chatty prose — no rollback, no house file format, nothing I can drop into migrations/. Next: few-shot examples.

V3 · Technique: Few-Shot Examples

+ one real migration file as format example
Prompt — change highlighted
[V2 prompt, plus:] Match the house format of my existing migrations. Example (a real file from the project): <example> -- Migration: 014_add_site_id_to_workers -- Author: Sara Arif | Date: 2026-03-02 -- Purpose: workers move between sites; payroll must know where hours accrued -- ==== UP ==== ALTER TABLE workers ADD COLUMN site_id INT UNSIGNED NULL AFTER cnic; -- ==== DOWN ==== ALTER TABLE workers DROP COLUMN site_id; </example>
Output — representative excerpt
-- Migration: 015_add_audit_log -- Author: Sara Arif | Date: 2026-07-15 -- Purpose: attendance/payroll changes must be traceable (who, when, -- old→new) to prevent labour disputes -- ==== UP ==== CREATE TABLE audit_log ( … as V2 … ); -- ==== DOWN ==== DROP TABLE IF EXISTS audit_log;
Note — what improved in the outputHouse style adopted exactly — the filename continues my sequence (015 after 014), the header matches field-for-field, and the UP/DOWN split gave me a rollback for free because the example demonstrated one. Worth recording: in my FL-05 ladder an example made prose output worse (it photocopied the example's facts). Here the same technique was the cleanest win. The difference: this example carries format; the facts still come from V2's schema — nothing in a migration header to contaminate. Examples steer structure safely; they steer content dangerously. Next: nothing verifies my FL-01 criteria → output structure.

V4 · Technique: Output Structure

+ four XML-tagged sections
Prompt — change highlighted
[V3 prompt, plus:] Structure the response in exactly four sections, using these XML tags: <migration> the UP/DOWN file, house format </migration> <verification> SQL I run before and after: row counts on every existing table, and how to time the migration </verification> <risks> what could go wrong on a live 10k-row database, each with its mitigation </risks> <assumptions> anything you assumed that I should confirm </assumptions> No text outside these tags.
Output — representative excerpt
<verification> -- BEFORE and AFTER — must match exactly: SELECT COUNT(*) FROM workers; SELECT COUNT(*) FROM attendance; SELECT COUNT(*) FROM payroll_periods; -- Timing: SET profiling = 1; … SHOW PROFILES; </verification> <assumptions> - changed_by comes from the PHP session user, passed by application code. - Trigger-based capture is a separate, later migration. </assumptions>
Note — what improved in the outputThe deliverable became complete and checkable — the verification queries directly test criterion (b), the timing method addresses (c), and the assumptions section surfaced two decisions I hadn't made consciously (where changed_by comes from; triggers deferred). The prose disappeared because the tags left it nowhere to live. Still failed: reasoning is shallow — generic risks, straight to SQL. It documented risks; it didn't let them shape the migration. Next: step decomposition.

V5 · Technique: Step Decomposition

+ think-before-SQL, self-check last
Prompt — change highlighted
[V4 prompt, plus:] Work through these steps in order, showing each: 1. Restate the schema delta you are about to make. 2. List everything that could break on a LIVE 10k-row MySQL 8 database (locks, FK violations on existing rows, defaults, replication) — before writing any SQL. 3. Only then write the migration, mitigating each risk from step 2. 4. Write the rollback. 5. Write the verification. 6. Self-check: state pass/fail against my three criteria — and say what you cannot verify from here.
Output — representative excerpt
Step 2 — what could break: - CREATE TABLE acquires only a brief metadata lock: safe. - BUT any future ALTER adding an FK column to attendance (10k rows) must be NULLable or backfilled. This migration deliberately avoids touching existing tables at all: audit_log references records logically (table_name + record_id), not by FK — so criterion (b) cannot be violated by design. … Step 6 — self-check: (a) syntax: valid MySQL 8, verifiable by dry-run on a copy. PASS (b) rows preserved: no existing table is touched. PASS by design (c) <500 ms: expected ~10–50 ms — but I cannot measure your hardware from here. Run the timing in <verification> to confirm. MEASURE
Note — what improved in the outputThe design itself improved, not just the wrapping. Step 2 caught the FK-on-live-rows trap and redesigned around it — logical references instead of foreign keys, so no existing table gets altered and criterion (b) becomes unbreakable by construction. In V4 that risk didn't even appear. And the self-check refused to overclaim: "I cannot measure your hardware from here" is exactly the honesty my criteria need. Remaining limit: the model reasons about 500 ms; only my machine proves it — which is why FL-01 classified this task "Collaborate with AI," not "Fully Automate."

3  Cross-Model — Final V5 Prompt on Claude vs ChatGPT

DimensionClaudeChatGPT
Tone Terse, spec-like; each step a labeled block, no filler between sections. Friendlier, chattier — "Great, let's make this production-safe!" and prose transitions between the tagged sections, despite "no text outside these tags."
Structure Followed the 6 steps and 4 XML tags exactly; self-check rendered as a criterion-by-criterion table. Merged steps 1–2 into one narrative block; half the verification landed inside <risks>. Mostly there, but needed re-sorting before filing.
Accuracy Stuck to scope: exactly the table specified, logical-reference design, correct MySQL 8 throughout. Also correct MySQL 8 — but added an unrequested index on changed_at and an extra session_id column "for good measure." Useful ideas; scope creep on a payroll database is how surprises happen.
Failure points Hedged honestly on the 500 ms criterion: "expected 10–50 ms; measure on your hardware." Overclaimed: "this will comfortably run in under 500 ms ✅" stated as fact — it cannot know that. On a criterion that exists because payroll can't be wrong, unearned certainty is the worse failure.
One thing ChatGPT did better: it volunteered a trigger-based capture design (AFTER UPDATE triggers on attendance writing to audit_log automatically) as a follow-up — a genuinely good architectural suggestion Claude never raised because it stayed strictly inside the spec. Off-spec here, but it went into my backlog as migration 016. Verdict: Claude's output went into the project as-is; ChatGPT's needed ten minutes of re-sorting and de-scoping. For production SQL, the literal instruction-follower wins; for "what should I build next," ChatGPT's tangent was the most valuable paragraph either model produced.

4  The Final Reusable Template — No Personal Context Left

For any schema-change task on a live database. Replace the {braces}; nothing else needed.
You are a senior {DIALECT + VERSION} DBA who writes migrations for live production systems and treats data loss as a career-ending event. CONTEXT — the live schema (excerpts): {paste the CREATE TABLE lines or column lists for every table involved} This is a production database: {scale — rows, users}. Why this matters: {the business consequence of getting it wrong — one sentence}. Constraints: {your hard criteria, e.g. all rows preserved; runs under N ms}. HOUSE FORMAT — match this example from my project: <example> {paste ONE real migration file — header comment, UP/DOWN sections, naming convention} </example> STRUCTURE — respond in exactly these sections, no text outside them: <migration> the UP/DOWN file, house format </migration> <verification> SQL to run before and after: row counts on every existing table, plus how to time the migration </verification> <risks> what could go wrong live, each with its mitigation </risks> <assumptions> anything you assumed that I should confirm </assumptions> PROCESS — work through these steps in order, showing each: 1. Restate the schema delta. 2. List everything that could break on the live database (locks, FK violations on existing rows, defaults, replication) — BEFORE any SQL. 3. Write the migration, mitigating each risk from step 2. 4. Write the rollback. 5. Write the verification. 6. Self-check against my constraints — and say explicitly what you cannot verify from where you sit.

Usage notes for a stranger: the example must be a format example (file layout), never a content example — format examples anchor structure; content examples leak their facts into your output (proven in my FL-05 ladder). Keep step 6's "say what you cannot verify" — it converts model overconfidence into a to-do list for you. And run <verification> yourself: the model reasons about your constraints; only your hardware proves them.
