Case 1 · DG Cement Payroll & Attendance System

Proof · Flagship · Paying Client
The problem

A cement contractor paid 50+ construction workers off paper registers — attendance handwritten at remote sites, retyped, totaled manually. Payroll ate most of the site manager's day every week, and workers disputed totals because nobody trusted the registers.

What I did

Built an offline-first PWA — the decision everything hung on, because the sites barely have signal. Attendance recorded on-device, synced when connected; PHP/MySQL, 28 API endpoints, audit logs on every change. The 26-day payroll formula I did by hand, not with AI — matched the site manager's manual calculation exactly, because payroll off by one rupee starts a labour dispute. My first sync design assumed longer connectivity windows than reality; I reworked it until a device could go a full shift offline without losing a record.

What came of it

In production, and I still support it today. 50+ workers logged daily • weekly payroll processing time down 70%+ • zero data loss during outages.

Next time

Log the before-state formally, so "70% faster" isn't the only number I can defend.

"Payroll used to take me the whole day — now it's done in a couple of hours, and there are no more arguments over attendance. And she didn't disappear after delivery." — Site Manager, DG Cement project (paraphrased; name withheld on request)

Case 2 · The AI Core

Work · Built in This Internship
The problem

Most "AI backends" are a chat wrapper: prompt in, text out, hope for the best. The task: a backend that behaves like production software — answering questions over a private database of 14 research papers, where the LLM decides which tools to run, including writing its own SQL.

What I did

Treated the model as untrusted input, same as any user. Every SQL query it writes passes a guardrail: comments stripped, forced to start with select, locked to one table, 17 blacklisted patterns — no mutations, no pragma, no schema introspection, no chained statements. Output validated against a schema; a non-conforming answer returns an explicit 422 instead of passing garbage downstream. Provider-agnostic via gateway: Gemini, Groq, or Claude swapped with one env variable.

What came of it

51 unit tests pass, 22 of them security tests — 16 malicious queries blocked. An internship build, not client work, labeled that way. What it shows: validate in, sanitize the middle, validate out.

Next time

Validate the SQL structurally (parse it, not pattern-match it) — keyword filters are a first layer, not a wall.

Case 3 · AURA — LLM Chat Interface

Work · Depth
The problem

A chat interface where multiple conversation threads stay truly separate. With LangGraph and a SQLite checkpointer under concurrent access, they didn't — state bled between threads.

What I did

Built on LangGraph StateGraph with persistent SQLite checkpointing, then spent the hard part diagnosing the thread-isolation failure by hand — AI brainstormed edge cases, the concurrency verification was mine. Also made it survive the real world: provider 404s and rate limits trigger fallbacks instead of killing the chat.

What came of it

A working multi-threaded chat with persistent, isolated conversations. Runs locally, code on GitHub, I'm the only user — no client behind it; it lives in the portfolio as depth, labeled exactly that.

Next time

Load-test the checkpointer before building UI on top of it, not after.

Case 4 · DataSage — Multi-Agent EDA Pipeline

Work · Depth
The problem

Exploratory data analysis is the same loop every time — profile, find what matters, explain, chart. Doing it by hand per CSV is slow; most automated tools crash the moment the data is dirty.

What I did

A three-node LangGraph pipeline — analysis, narrative, visualization — passing structured JSON validated against the frontend's schema. Two decisions I'd defend: structured error states for empty/corrupted CSVs instead of crashes (I generated hostile test data — nulls, outliers — specifically to attack it), and the full run in under 15 seconds, because a tool people wait minutes for doesn't get used.

What came of it

Holds both targets on real data, not just my synthetic attacks — ran against public Kaggle datasets, stayed under 15s end-to-end, no crashes. Portfolio-depth build, honestly labeled.

Next time

A real business's data, where the mess isn't curated — Kaggle data is public and half-cleaned; a client's export is neither.

Case 5 · Inventory Management System

Work · Ops
The problem

Small shops run stock the way the cement contractor ran payroll: counts in a register, sales in someone's head, and nobody notices an item is out until a customer asks for it. Built as a portfolio build modeled on how a real shop operates — labeled that way.

What I did

A desktop system in C# on SQL Server. The cashier gets exactly one screen — the POS; every sale decrements stock automatically. The admin gets everything else: low-stock alerts, PDF/CSV reports, and an audit log on every stock change — who touched it and when — because unexplained stock movement is the whole problem. Two decisions: role-based access (a cashier who can edit inventory is an audit hole, so they can't), and an AI chatbot that answers "what's in stock?" in plain language and auto-fills restock entries when something runs out.

What came of it

A complete working system: POS, alerts, audit trail, reports, chatbot. No paying shop behind it yet — the build I point an ops manager to when they ask "can you do inventory, not just payroll?"

Next time

Put it in one real shop for a month before calling it done — the register it replaces will teach me things my test data can't.

Case 6 · Customer Trends ETL

Work · Data
The problem

Raw customer data doesn't answer questions — it's nulls, inconsistent formats, and duplicates pretending to be a dataset. I wanted the pipeline that turns that mess into a trend you can act on, end to end.

What I did

An extract-transform-load pipeline over a public customer dataset: cleaned and standardized the raw data (nulls handled explicitly, formats normalized, duplicates resolved — the unglamorous 80%), loaded into a queryable store, trend analysis with visualizations on top. The decision that mattered: cleaning as its own auditable stage — when a number looks wrong downstream, I can point to exactly what the transform did to it.

What came of it

A working pipeline from raw file to trend charts. Public data, so I claim the engineering, not a business outcome: the same pipeline pointed at a client's sales export is how "which customers are slipping away?" gets answered.

Next time

Run it on data someone actually needs an answer from — trends only matter when a decision hangs on them.

Case 7 · Mobile Expense Tracker

Work · In Daily Use
The problem

My own money was leaking the way most students' does — small expenses, untracked, remembered wrong. Notes apps don't add up columns, and every tracker app I tried wanted an account and a connection.

What I did

Built my own: fully offline, data stored on-device. The same instinct as DG Cement — if the tool needs a connection to record a 50-rupee entry, it won't get used, and expense data is nobody's business but mine anyway.

What came of it

It's on my phone and I log expenses in it regularly. One real daily user — me — which is one more than most portfolio apps get. The smallest demonstration of the pattern I sell: offline-first, on-device, built around how the person actually behaves.

Next time

Add an export — my own data is now locked on one device, the exact problem I build my way out of for clients.

Case 8 · Global Sales Dashboard

Work · Data
The problem

A global sales spreadsheet can tell you the total. It can't tell you at a glance which region carries the profit, which product line drags, or where the numbers head next quarter — you'd be pivot-tabling for an hour per question.

What I did

A Streamlit dashboard with Plotly charts over a public global sales dataset. The design decision that mattered: interactive filters — year, region, category — so the reader interrogates the data instead of scrolling a report. One step past describing the past: time-series forecasting, so it projects where sales are heading, not just what sold.

What came of it

One screen that answers the region/product-mix question the spreadsheet couldn't, with a forecast attached. Public data, so the claim is the capability: what "reports" means beyond a PDF.

Next time

Validate the forecast against a held-out period and show the error on the chart — a projection without its error bar is a guess in a suit.
